# III WEAVE CENSUS — Phase 1 (read-only) — THE DECISIVE FINDING: III HAS NO DEAD CODE

Read-only scan of all `STDLIB/iii/**/*.iii` + the `STDLIB/corpus/` references. **Nothing changed.**

## The finding that changes the plan

Of **678** modules:

- **377** are LOAD-BEARING (imported by another module) — the spine.
- **296** are CORPUS-TESTED CAPABILITIES — *not* imported by another module, but each
  has a corpus KAT that references it. These are **tested, intentional islands**, not dead code.
- **5** are ENTRY-POINT / governance (compiled/run by the build, not imported).
- **0** are TRULY UNREFERENCED anywhere (no importer, no corpus ref).

**There is no dead code to cut.** This is not luck — III's *coverage ratchet* (`project_iii_coverage_ledger`:
every `@export` is KAT-proven and reachable from a corpus use-site) **forbids** an unreferenced export by
construction. So the perceived 'bloat' is **not** dead modules — it is the **absence of a self-model**
(296 capabilities tested in isolation, never wired together so the system *knows* them) and possibly a few
**superseded duplicates** (a newer module does X better; the old X is still tested but redundant).

**Therefore the plan's deletion phases have ~zero safe targets, and mass-deletion is OFF the table.**
The honest, livelihoods-safe value is: (1) the **self-model weave** (so III knows its 678 capabilities and
their relations), (2) **superseded-duplicate consolidation** (merge, gated, only where one truly supersedes
another — found by capability-hash in Phase 2, *not* by name), (3) **production-grade standardization** of
the kept code (most already K≥0.99), (4) **un-orphaning** genuine gaps, (5) the **frontier**.
